# Face Scanner to VR Avatar Pipeline

## 概要
本プロジェクトは、iPhoneのTrueDepthカメラを利用して顔の形状（メッシュデータ）と表情をスキャンし、Mac（PC）上のローカルサーバーで解析・結合を行い、自動的にVR空間用のアバター（GLB形式）を生成してUnityクライアントに直接ロードするまでの**「完全自動化パイプライン」**です。

## 全体の処理フロー
1. **iOSアプリ（スキャン＆送信）**
   - ユーザーがiPhoneに顔を向け、指示に従って顔を回す（8方向のスキャン）。
   - 「あいうえお」の発音動画とブレンドシェイプ（表情データ）を取得。
   - 顔の立体メッシュ（OBJ）と動画データをPC（GPUサーバー）のFastAPIへ送信。
2. **バックエンドサーバー（解析とGLB組み立て）**
   - FastAPIがiOSからのメッシュ・動画・JSONデータを非同期で受信。
   - OpenCV等を用いて動画から最大開口時（口の開き）を解析し、最適な「歯・口内」モデルを抽出。
   - Trimeshを用いて、iPhoneから送られた顔面メッシュ、ベースとなる後頭部メッシュ、生成した歯のモデル、髪型アセットを3次元空間で位置合わせし、結合。
   - 完全なVRアバターとして1つの `avatar.glb` ファイルに出力。
3. **Unityクライアント（自動ダウンロード＆表示）**
   - バックグラウンドで生成の完了状態をポーリング監視。
   - 生成が完了した瞬間にGLBファイルをダウンロードし、`UniGLTF` を介してゲーム実行中（ランタイム）にVR空間へ動的にロード・出現させる。

## 既存システムの活用と独自開発の切り分け

本システムは、優れた既存の基盤技術の上に、アバター生成特有の課題を解決するための独自のロジックを開発・統合して構築されています。

### 🛠 活用している既存システム（基盤）
- **ARKit (Apple)**: 高精度な顔のデプスメッシュ取得と表情（Blendshapes）のトラッキング。これをゼロから自作するのではなく、iPhoneのハードウェアパワーを最大限に活用。
- **FastAPI (Python)**: 高速かつモダンな非同期通信API。重いファイルのアップロードやダウンロードを効率的に処理する土台として利用。
- **Trimesh / OpenCV (Python)**: 3Dメッシュの頂点計算と画像解析のためのライブラリ。
- **UniGLTF (Unity)**: GLBファイルをランタイムでパースし、UnityのGameObjectに変換する基盤機能。

### 🚀 独自で開発した部分とその目的（なぜ作ったか）

1. **iOS側の「放射状（Radial）角度スキャン判定」と「8分割リングUI」**
   - **目的**: 既存のARKitサンプルでは「ユーザーが今どの角度を向いていて、あとどこを向けばいいのか」が視覚的に全くわからず、操作性が非常に悪かったため。
   - **開発内容**: FaceIDのセットアップ画面のような「円形の8分割ゲージ」をゼロから実装。また、首の自然な円運動に合わせた独自の「放射状角度計算アルゴリズム」を実装し、斜め方向でもスムーズに判定されるようUXを劇的に改善した。

2. **メッシュ自動結合パイプライン（アバターアセンブリ）**
   - **目的**: iPhoneから取得できるのは「顔の表面（お面のような状態）」のデータのみであり、そのままではVRアバターとして後頭部や口の中がスッポリ抜けてしまい使い物にならないため。
   - **開発内容**: 送られてきたお面状のメッシュに対し、ベースとなる頭部モデル、個別の歯、選んだ髪型を座標計算によって自動で縫い合わせ、1つの完全なGLBモデルに統合する処理を独自に構築した。

3. **Unityの非同期フェッチャー（`AvatarFetcher.cs`）**
   - **目的**: 3Dモデルの生成処理はサーバー側で数秒〜数十秒かかる重い処理であるため、生成を待っている間にUnityアプリ全体がフリーズ（画面停止）してしまうのを防ぐため。
   - **開発内容**: Unityのメインスレッドを阻害せず、裏側でサーバーの生成完了を監視（ポーリング）し、データが完成した瞬間に自動でダウンロードして空間にポップさせる独自の通信ロジックを実装した。

## 今後のロードマップ（開発予定）

本システムは、MVP（静的メッシュの結合）の検証を完了し、現在以下の「完全版アバター」に向けたアーキテクチャへの移行を進めています。

1. **表情（Blendshapes）のフル稼働対応（★現在着手中の最優先タスク）**
   - **課題**: 現在のTrimesh結合では、iPhoneから取得した「静的なメッシュ」を結合しているため、表情を動かすためのモーフターゲット（Blendshapes）が含まれず、彫刻のように動かないアバターになってしまう。
   - **解決策**: ARKitの52種類のBlendshapeが事前実装された「アニメテイストのベーステンプレートモデル」を用意。iPhoneから送られた顔の輪郭データに合わせてテンプレートモデルを変形（Morph）させる**「テンプレート・マッピング方式」**へ移行し、表情が100%完璧に動くアバターを生成する。

2. **リアルな顔写真テクスチャの適用**
   - **課題**: 現在は形状データ（OBJ）のみを取得しているため、肌が単色のクレイモデルになっている。
   - **解決策**: iOSのARKitからスキャン時に「顔写真（カメラフレームのJPEG画像）」を同時に取得し、サーバーへ送信。サーバー側でその写真をベースモデルの顔にUVマッピングして焼き付け、写真ベースのリアルな顔立ちを再現する。
